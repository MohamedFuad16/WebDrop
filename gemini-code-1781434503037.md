# Technical Specification: Cross-Platform Web-Native File-Sharing Engine (MVP)

This document specifies the end-to-end architecture for a decentralized, memory-safe, cross-platform file-sharing web application. The engine operates entirely within modern mobile and desktop browsers (iOS Safari & Android Chrome) without requiring native app wrappers, plugins, or external installations.

---

## 1. System Topology & Architectural Principles

The system splits network operations into two entirely isolated layers: the **Signaling Pipeline** (orchestration) and the **Data Pipeline** (transportation).



### The Core Separation Rules
1. **The Handshake / Signaling Layer:** Managed via stateful WebSockets over an NGINX reverse proxy. This lane handles text metadata only (handshakes, presence, connection offers, proximity telemetry). It **never** touches binary file data. Firewalls allow this easily because it is an *outbound connection up to a public cloud server*.
2. **The Data Transport Layer:** Managed via WebRTC `RTCDataChannel`. Data flows directly between browsers over an encrypted, peer-to-peer (P2P) bridge. 
3. **The Relayed Transport Fallback (TURN):** Triggered automatically **only** when firewalls prevent a direct peer-to-peer path.

---

## 2. The Network Obstacle: Symmetric NAT & Carrier-Grade NAT (CGNAT)

The primary engineering hurdle in real-time browser networking is crossing firewalls, specifically **Symmetric NATs** (found on corporate/hotel Wi-Fi) and **Carrier-Grade NATs (CGNAT)** (enforced globally by 4G/5G mobile carriers).



### How Home NAT (Cone NAT) vs. Symmetric NAT Behaves
* **Home Wi-Fi Routers (Cone NAT):** Highly permissive. When a device checks its public mapping using a **STUN server**, the router opens a public mapping (`Public IP : Port`). *Anyone* on the internet can send data back to that specific port, allowing a direct P2P connection to succeed for free.
* **Mobile Carriers & Corporate Wi-Fi (Symmetric NAT / CGNAT):** Strictly secure. The router creates a unique public mapping **per destination**. If your phone talks to a STUN server, the router assigns `Port 5000`. The moment your phone tries to connect directly to a peer's phone, the router secretly switches your connection to a completely different, random port (e.g., `Port 7821`) and **instantly drops** any incoming traffic from that peer trying to reach `Port 5000`.

## 3. WebRTC Data Infrastructure & Server Economics

Since mobile data conditions regularly enforce TURN routing, selecting the right server infrastructure is critical to protecting operational budgets from high cloud bills.

### The Network Infrastructure Breakdown
* **`coturn` (Self-Hosted Open Source):** The industry-standard STUN/TURN server written in C. It is open-source, highly performant, and completely free to use.
* **Managed TURN (SaaS):** Paid, third-party global Anycast networks (e.g., Twilio, Cloudflare Calls, Metered.ca). They charge metered rates per gigabyte to offload infrastructure management entirely.
* **Cheap Egress Relays (The Cost Hack):** Major cloud providers like AWS, GCP, and Azure allow free incoming data (**Ingress**) but charge exorbitant fees (~$0.08–$0.09 per GB) for outgoing data (**Egress**). Because a TURN server acts as a relay, every file chunk counts as both Ingress and Egress, generating massive bills on AWS:

$$\text{1,000 file transfers} \times \text{1 GB payload} \times \$0.09 \text{ egress rate} = \$90.00 \text{ per TB}$$

**The Solution:** Deploy `coturn` on alternative flat-rate infrastructure providers (such as Hetzner, OVH, or Linode) that bundle massive monthly data allocations (e.g., 20 TB) into flat, predictable monthly server fees ($5–$12/month).
