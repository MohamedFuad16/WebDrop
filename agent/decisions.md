# Architecture Decision Records (ADRs)

Append-only. Newest at the bottom. Format: ID · date · title · status · context · decision · consequences. These initial ADRs are **reverse-engineered** from the existing architecture to capture the rationale evident in the code.

## ADR-0001 · 2026-06-29 · Use WebRTC data channels for file transfer
**Status:** Accepted (in code)
**Context:** WebDrop transfers user files (up to 500 MB) between two devices that are physically near each other. Routing file bytes through a central server is costly, slow, and a privacy concern.
**Decision:** Send files **peer-to-peer over WebRTC `RTCDataChannel`** (`js/services/webrtc-transport.js` + `data-channel-transfer-protocol.js`). The WebSocket server is used only for the control plane (presence, invites, SDP/ICE relay, telemetry); it explicitly **rejects binary frames**. Two ordered channels separate control (`webdrop-control-v1`) from file bytes (`webdrop-file-v1`), with a manifest+chunk+ack protocol, SHA-256 integrity, backpressure, and retries.
**Consequences:** Direct/relay path depends on NAT and TURN availability (see `errors.md`); needs a TURN broker for symmetric NATs. Keeps bytes off the server and scales the data plane horizontally for free. Adds protocol complexity (ordering, retry, caps) the app must own.

## ADR-0002 · 2026-06-29 · Provide a mock signaling + simulated runtime fallback
**Status:** Accepted (in code)
**Context:** Real signaling, proximity (ultrasound/motion), and transfer require a deployed server, real hardware, and two devices — terrible for UI iteration and CI.
**Decision:** Define one signaling interface implemented by both `WebSocketSignalingAdapter` (production) and `MockSignalingAdapter` (in-memory, 15 fake peers, simulated invites/proximity/QR/RTC). Runtime flags (`runtime-flags.js`/`runtime-config.js`) select the adapter and gate `realProximityCeremony`/`realTransfer`/`qrPairing`; `?runtime=mock` forces mock on localhost.
**Consequences:** The entire UI runs offline with zero backend. Risk: developers may think they're hitting production when they're on mock (documented in `errors.md`). Mock and real paths must be kept behaviorally aligned.

## ADR-0003 · 2026-06-29 · Multi-factor physical-proximity verification (ultrasound + motion, QR fallback)
**Status:** Accepted (in code)
**Context:** "Nearby" must mean physically present, not merely on the same signaling server, to prevent remote impersonation/relay attacks during pairing.
**Decision:** Run a **ceremony** combining ultrasonic chirp exchange (`acoustic-proximity.js`), DeviceMotion bump+tilt (`motion-proximity.js`), and an optional QR token (`proximity-token.js`), scored by `proximityScore`. A pass requires score ≥ 55 **and** ultrasound+bump+tilt present (`hasRequiredPhysicalEvidence`), mirroring the authoritative server gate. QR pairing is the explicit fallback when sensors are unavailable.
**Consequences:** Strong proximity assurance; but hardware/permission/secure-context dependent and brittle on some devices (see `errors.md`). The client must not over-report success vs the server, hence the duplicated gate.

## ADR-0004 · 2026-06-29 · Ship as a no-build static PWA with a layered storage ladder
**Status:** Accepted (in code)
**Context:** The app targets mobile browsers (esp. iPhone) and must be cheap to host, installable, offline-capable, and able to save large received files despite divergent browser storage support.
**Decision:** No bundler/transpiler — plain ES modules served statically (`scripts/static-server.mjs`), versioned via `?v=<APP_VERSION>` and a precaching `service-worker.js`. Received files use a capability/size-selected backend ladder in `StorageClient`: OPFS/IndexedDB (deferred) → StreamSaver streaming → in-memory Blob (iOS, ≤128 MB).
**Consequences:** Zero build pipeline, trivial deploy, instant module reloads — but no tree-shaking/minification and manual cache-version discipline (bump in several places). Storage behavior differs by platform; iOS large transfers are capped.

## ADR-0005 · 2026-06-29 · Render the architecture diagram with D2's ELK engine
**Status:** Accepted
**Context:** `graph/architecture.d2` was rendered with D2's default `dagre` layout, which routed connectors straight through boxes and swept long curved edges across the canvas (notably the `transfer → transport.protocol` `sendFiles` edge), making the diagram hard to read.
**Decision:** Render with the **ELK** layout engine (orthogonal edge routing, avoids node overlap) — applied both in the source (`vars.d2-config.layout-engine: elk`, `pad: 40`, `center: true`, `direction: down`) and via the CLI (`d2 --layout elk --pad 40`). Adopt clean-layout authoring rules: containers ordered top-to-bottom in data-flow order, tightly-coupled nodes in the same container, hub node (`controller.js`) kept central, short single-line labels, no manual `near` constraints, and nodes repositioned (e.g. `transfer-engine.js` next to the transport nodes, admin next to Azure) rather than crossing the canvas with long edges. ELK is open-source and bundled with d2; the proprietary `tala` engine was rejected.
**Consequences:** The diagram is clean — no connector crosses a box, no containers overlap. The same change was baked into the `agent-folder` skill (render script + template + docs) so future diagrams render cleanly by default. Authors must keep `.d2` sources flow-ordered for ELK to produce the best layout.

## ADR-0006 · 2026-06-30 · Personalize the connection QR with an avatar badge (level H)
**Status:** Accepted (in code)
**Context:** The "Show this code nearby" QR was a generic coloured matrix. Product wanted it to feel personal (per-user) without hurting scannability or adding latency to the pairing ceremony.
**Decision:** Render the QR at **error-correction level "H"** (~30% recovery) instead of "M", then composite the user's avatar as a rounded centre badge sized to ~24% of the code over a white "knockout" quiet zone (`DynamicIsland.drawQr(token, avatar)`). Avatar images are decoded once and kept in a module-level `qrLogoCache`, so a warm badge paints synchronously in the same frame as the modules — no async gap before a scan. A node test (`tests/qr-personalized.test.mjs`) reproduces the exact geometry into an RGBA buffer and asserts `jsQR` still decodes the original token, so the personalization can never silently break pairing.
**Consequences:** The code is denser at level H (smaller cells for the same token) but still scans comfortably at the rendered size, and the centre logo is well within H's recovery budget. Any future change to badge size / knockout padding must keep the decode test green. The same `drawQr` path is used only for the *display* QR; the scanner path is unaffected.

## ADR-0007 · 2026-06-30 · The transfer HUD is the tile-wave itself (no separate bar)
**Status:** Accepted (in code)
**Context:** During a transfer the island showed an animated tile-wave dimmed to 30%/blurred behind a separate gradient progress bar (`.webdrop-island__meter`). It felt redundant and busy — two competing progress visuals on a tiny surface.
**Decision:** Make the tiles themselves the progress bar. `TileWave` gained `setTransferMode(on)` + `setProgress(ratio)`; in the draw loop each tile computes `colFill = (sweepCol + 1) / cols` (where `sweepCol` already mirrors for send vs receive via `setDirection`). When transfer mode is on and `colFill > progress` the tile renders as a dim, desaturated "remaining track" with no glow; filled tiles keep the normal animated sweep, so the lit region grows with progress while still shimmering. The static/reduced-motion path (`#drawStatic`) honors the same rule, and `setProgress`/`setTransferMode` trigger a redraw when the rAF loop isn't running (`#repaintIfStatic`). `DynamicIsland.paintTransferProgress` now calls `wave.setProgress` and only updates the percent text; the `.webdrop-island__meter` element + CSS were deleted and the wave stays full-opacity during transfer (`.webdrop-island__transfer` is a full-height flex column — percent/label top, filename bottom, tiles through the middle).
**Consequences:** One coherent progress visual; less DOM/CSS. The percentage text now sits over the tiles, so it carries a text-shadow for legibility. Removing the bar required updating the `app-ui` e2e (it asserted on `[data-island-transfer-bar]`); the `data-island-transfer-percent/label/name` hooks are unchanged.

## ADR-0008 · 2026-06-30 · Propagate live profile changes over the data channel (+ best-effort presence)
**Status:** Accepted (in code)
**Context:** Changing avatar/ring/name only mutated local `state.self`; an already-connected peer (and nearby devices) kept showing the stale icon/name until reconnect.
**Decision:** Reuse the existing control plane. The connected case is reliable: the controller broadcasts `{ type: "profile:update", profile: { name, avatar, avatarId, ringColor } }` over the open WebRTC **control** channel via a new `WebRtcTransport.sendControlMessage()` → `DataChannelTransferProtocol.sendControl()`; the protocol's `default` switch case already emits a `control` event, which the transport now forwards. The controller validates the incoming profile (`isAllowedAvatarChoice`, `#ringColor` regex, 22-char name), patches the connected peer in the store (re-renders orbit + connected screen), and calls `DynamicIsland.updatePeerProfile()` to refresh the island peer avatar/name live (the store-driven `sync` can't, since the peer id is unchanged). The nearby case is best-effort: `WebSocketSignalingAdapter.updateProfile(self)` re-sends `client:hello` with the cached `capabilities`, no-opping when the socket is closed (`MockSignalingAdapter` just updates its cached self). Sends are debounced 250 ms with change-detection so name typing doesn't spam.
**Consequences:** Connected peers update instantly without reconnect; nearby refresh depends on whether the (external) relay rebroadcasts the re-announce. Custom (data-URL) avatars are allowed through and travel over the ordered control channel — acceptable for the small 256² crops, but very large data URLs would be a heavier control message.

## ADR-0009 · 2026-07-01 · Persist and snapshot runtime proximity policy; record tests against revisions
**Status:** Accepted (in code)
**Context:** Real users rarely tap Connect in the same instant. The prior effective admission window was only about 3.6 seconds (1.8-second join window plus one extension), while ceremony timing and scoring were spread across environment variables and constants. Tomorrow's multi-person test also needs reproducible evidence: changing a threshold halfway through a session, or losing which values produced a result, would make calibration unreliable.
**Decision:** Make one validated `RuntimeProximityPolicy` authoritative for score weights, minimum score, late-tap grace, acoustic window, and match slop. Persist it atomically on the signaling server, expose it through authenticated `GET`/`PUT /api/proximity-policy`, and snapshot the current revision onto every new session. A lone first tap may remain admitted for the full grace (default 6,000 ms); once a partner joins during the extension the cohort starts after a short 300 ms settle. Active sessions never mutate. The client receives the snapshot in `proximity:start` and applies the same scoring/timing. Admin tuning lives in **Settings**, separate from the live monitor. Test recordings store assignments, bounded diagnostics, derived outcomes, and the exact policy snapshot locally on the admin device.
**Consequences:** Operators can tune and refresh without redeploying, and every saved run remains interpretable. Weights must total 100; server bounds prevent unsafe windows. Policy changes affect only new sessions. The global cap remains 100 participants (up to 50 pairs), while the configured six-device cohort cap stays conservative even though the longer acoustic window has a higher theoretical slot ceiling. Admin run history is browser-local and is not a cross-operator database.

## ADR-0010 · 2026-07-02 · Admin readiness reflects real health/proof; dashboard is mobile-first and bilingual
**Status:** Accepted (in code)
**Context:** Operators run the admin dashboard from phones during physical test days and share it with several helpers. Three problems surfaced: (1) the dashboard showed a false "22% ready / server offline" whenever the browser had no admin token, because the token-gated `/api/diagnostics-public` snapshot failure cascaded into `serverReachable = false` even though the public `/readyz` was healthy — 22% is literally the server-unreachable branch (2 of 9 checks); (2) the "readiness %" ignored that 1-on-1 proximity pairing already works in live runs; (3) the desktop Settings form was a cramped 3-column grid and the Test cases tab overflowed horizontally on phones, and the test-case catalog was English-only. Auth is a single shared `METRICS_API_TOKEN` compared by exact string match on the server, so distinct per-user tokens are impossible without a server change + VM deploy — out of scope for the test day.
**Decision:** (A) Separate reachability from authorization in `refreshDiagnostics`: the public `/readyz` and proximity-policy reads decide `serverReachable`; a missing/invalid token only withholds the authenticated live snapshot and surfaces a non-alarming `adminTokenNeeded` banner, never a false offline. (B) Represent proven-but-partial capability with a new **"partial"** readiness status (proximity ceremony = "1-on-1 verified", ◐) that counts in `readinessSummary`, yielding a truthful 56% on a healthy server rather than a hardcoded figure. (C) Make the dashboard mobile-first: 2-column Settings fieldsets + full-width action row, ≥9.5px font floor for helper text, and `.test-main { grid-template-columns: minmax(0,1fr) }` + `.test-main > section { min-width:0 }` to stop grid children from forcing overflow. (D) Localize the 7-case test catalog into Japanese via a `TEST_CASE_I18N_JA` override map + `localizeCase()` while keeping `test-runs.js` (English) as the single source of truth. Keep the single shared token; the operator distributes it to helpers over a secure channel.
**Consequences:** Tokenless viewers (helpers who haven't pasted the token yet) now see accurate server health and clear guidance instead of a scary false-offline. The readiness number tracks real progress and will rise as multi-device/two-pair proof lands (flip the proximity item off "partial"). No backend change: per-user tokens/revocation would require adding a token list to `hasMetricsAuthorization` and redeploying the Azure VM. Cache correctness required bumping 1.0.93→1.0.94 across all `?v=` suffixes, both `APP_VERSION` constants, and the package manifests together (see `errors.md`).

## ADR-0011 · 2026-07-02 · Service worker must not touch the cross-origin API; admin console restructured for live testing
**Status:** Accepted (in code)
**Context:** From an emulated iPhone view the deployed admin page showed a false "Offline" with a `NetworkError`, while desktop/curl worked. Root cause: the service worker's generic `fetch` branch intercepted **cross-origin** requests too — every call to the Azure signaling server (`/readyz`, `/api/proximity-policy`, `/api/diagnostics-public`, and by extension the live channel) — and lacked error handling, so any transient failure (notably during the SW update that follows a version bump) surfaced as a failed fetch and, via `refreshDiagnostics`, flipped the whole page to "unreachable". CORS was verified correct (vercel + localhost origins get `Access-Control-Allow-Origin`, preflight 204). Separately, the console needed real-use ergonomics: the summary strip showed on every tab; the readiness board was a wide non-collapsible grid with an "eye"-looking status glyph; single-device signals were mixed with pairing concepts and a duplicate "Active proximity sessions" panel; timing tuning used bare number fields with no guidance; and points showed decimals.
**Decision:** (A) The service worker early-returns for any request whose origin ≠ `self.location.origin`, so cross-origin API/WS traffic bypasses the SW entirely (native fetch, like curl); the same-origin fallback branch also gained a cache `.catch`. (B) `isServerHealthy()` additionally trusts a live WebSocket (`isSocketLive()`) as proof of reachability so a transient HTTP hiccup can't show a false Offline. (C) The summary strip lives inside the Readiness panel only. (D) The readiness board is a vertical accordion (`state.readinessCollapsed`) with clearer icons. (E) Live testing is a vertical stack: Connected devices carries an inline pairing badge (`deviceSessionState`, replacing the separate sessions panel), a Single-device test, a new Two-device test (`renderMultiDevice` — two devices side-by-side + live pairing outcome from session data, reusing the existing monitor telemetry), and an event timeline showing the latest 4 with a show-more toggle. (F) Settings timing windows are slider + number box with plain-language explanations and recommended/allowed range notes; weight total and minimum score are integer and aligned. No server/policy change (min score stays 55, revision 2).
**Consequences:** Cross-origin diagnostics are now immune to SW-induced failures, which was the real fix for the false Offline; the WS-reachable fallback is defense-in-depth. The two-device panel is pragmatic — it reuses per-device monitor telemetry plus whatever the proximity session exposes, so full reciprocal detail only appears once a real ceremony runs between the chosen pair. Some old session-render helpers are now dead code (left in place; safe to prune later). Cache-affecting → bumped 1.0.94→1.0.95 across all `?v=`, both `APP_VERSION`, and manifests.

## ADR-0012 · 2026-07-02 · Single-device test drops reciprocal-audio rows; two-device test shows full per-device panels
**Status:** Accepted (in code)
**Context:** Operator feedback on the v1.0.95 console: (1) the single-device test still listed "Heard signal / Correlation / Energy margin", but a lone phone in the monitor emits *and* listens, so those rows only reflect it hearing its own chirp — misleading self-noise, not a proximity result; (2) the two-device test's compact 4-row columns were too thin — the operator wants each phone's full live frequency map and complete signal readout side-by-side, with the pairing outcome; (3) the connection state was shown twice (top-bar "Connected" label + summary strip/inline status); (4) Settings looked cramped: timing windows were a 3-column grid with tiny text, the weight total had a stray `border-left`, and the minimum-score control wrapped to its own line; (5) test cases had no visual for how to physically hold the phones; (6) the "Live updates" toggle and a "refreshes every second" footer line were noise; (7) the session-table render helpers left dead by ADR-0011 were still present.
**Decision:** (A) Refactor the monitor renderers into pure builders — `frequencyChannelsHtml()`/`frequencySpectrumBlockHtml()` and `metricDefinitions()/metricTableHtml(telemetry, mode)` — and tag reciprocal rows `pairOnly`. The single-device table renders `mode:"single"` (Emitted packet, Sample rate, Bump, Tilt only); each two-device column renders `mode:"pair"` as a full panel (frequency map + all rows, compacted to metric·current·status via CSS). `latestSessionEvidence/latestEventEvidence` gained an optional deviceId so per-column evidence resolves correctly. (B) Removed the top-bar connection label; the single live `[data-server-connection]` chip now lives in the Live devices heading and is driven by `renderSummary` (guarded for absence). (C) Polling is always on — deleted the `data-live-poll` toggle. (D) Settings: `.score-summary` row aligns total + minimum on one line, `border-left` removed; timing windows are single-column cards; policy-revision display removed; primary button relabeled "Save & apply" (unchanged behavior — `updateProximityPolicy` applies to the signaling server instantly, no Vercel redeploy). (E) Added inline SVG hold-diagrams (`phoneSvg`/`testCaseIllustration`) per case. (F) Purged the dead session-table cluster + orphaned i18n keys. No server/policy change (min score 55, revision 2).
**Consequences:** The single-device test now only asserts what one phone can truthfully prove; the two-device test is the sole place reciprocal audio is judged. The two-device columns still depend on live per-device monitor telemetry + the proximity session, so full data appears only once both phones are monitored and a ceremony runs. Score-weight changes still apply server-side on Save with no code deploy; a *code* change (like this one) is what needs the Vercel push + version bump. Cache-affecting → bumped 1.0.95→1.0.96 across all `?v=`, both `APP_VERSION`, and manifests.
